## 1. Что такое KASTOL

**Минималистичный пульт управления ИИ-агентами**, которые работают на ПК пользователя и решают любые задачи (код, ресёрч, рутина), с подключением через десктоп / Telegram / голос.

**Сценарий:** юзер описывает задачу → визард настройки (провайдер, модели, команда агентов, рабочая папка) → лёгкая модель-планер выдаёт план → юзер ревьюит/правит → жмёт Run → команда агентов с разными характерами спорит и пишет код → юзер видит каждое сообщение, может вмешиваться (inject, reply, global rules, fork) → на выходе готовый результат в папке + полный лог.

**Главное отличие от CrewAI/AutoGen/Cursor:** мы **UX-обёртка** + **локальный** + **мульти-фронт** (web/TG/голос/десктоп) + **MCP-первоклассно** + **долговременная память** про юзера и его папки.

## 3. Стек (выбран — не меняем без согласования)

**Backend:** Python 3.11 + FastAPI + WebSockets + LangGraph + LiteLLM + SQLModel/SQLite + Chroma (RAG) + MCP Python SDK

**Frontend:** Next.js 14 (App Router) + TypeScript + Tailwind + shadcn/ui + Zustand + Framer Motion + Lucide + next-intl + next-themes

**Шрифты:** Inter (UI) + JetBrains Mono (код)

**Локализация:** EN дефолт, RU локаль с первого дня (i18n каркас в Phase 1)

## 4. Правила работы (нерушимые)

### Архитектурные
- **Каждая фича = новый файл, не правка старого**
- Новый tool = файл в `backend/app/tools/` с декоратором `@tool`
- Новый агент = JSON-файл в `backend/app/agents/personalities/`
- Новый workflow = файл в `backend/app/workflows/`
- Новый MCP-сервер = JSON-конфиг
- Новый плагин = папка в `~/.kastol/plugins/<name>/` с `manifest.json`
- Конфиги — JSON/YAML, без рекомпиляции
- Логи подробные с первого коммита

### Разработка
- **Каждая фича — отдельная ветка + отдельный PR**
- Имена веток: `feat/<short-name>`, `fix/<short-name>`, `chore/<short-name>`, `docs/<short-name>`
- Коммиты — Conventional Commits (`feat:`, `fix:`, `chore:`, `docs:`)
- **Тесты не пишем** без явной просьбы (пользователь — вайб-кодер)
- Зависимости — минимум, каждая обоснована

### Стиль кода
- Python: `ruff` + `black`, type hints везде, docstrings на публичных функциях
- TypeScript: `prettier` + `eslint`, строгий tsconfig
- Имена функций/переменных — английский
- Комментарии в коде — английский (короткие, по делу)

### Безопасность
- Local-first, никакой телеметрии
- API-ключи только в `.env` или native keychain
- Опасные tools (shell, computer use) — с подтверждением
- Watchdog на циклы, валидаторы tool-ответов, retry на API ошибки

### UX / Дизайн
- См. раздел 17 в `PROJECT.md` (полная дизайн-система)
- Чёрно-белая палитра, цвет только когда несёт смысл
- Один шрифт UI — Inter, моно — JetBrains Mono
- Анимации тонкие, тайминги 80/150/220/400ms, без bounce
- Никаких spinners — только skeleton или текстовый прогресс

## 6. Как со мной работать

### Когда я даю задачу
1. Прочитай `PROJECT.md` (если ещё не читал в этой сессии)
2. Прочитай раздел "Текущий статус" выше
3. Если задача неясна — **задай 1-3 уточняющих вопроса** прежде чем кодить
4. Если задача большая — **предложи разбить на PR-ы** прежде чем кодить
5. Покажи мне план перед тем как делать
6. Делай в отдельной ветке
7. После каждого логического шага — пиши краткий статус ("сделал X, теперь делаю Y")

### Когда что-то непонятно
- Не выдумывай. Спроси.
- Если решение спорное — предложи 2-3 варианта с pro/contra и попроси меня выбрать.

### Когда нашёл проблему в плане
- Останови работу, скажи мне. Возможно надо обновить `PROJECT.md`.
- Не "молчаливо" принимай свои решения которые расходятся с планом.

### Когда обновляешь архитектуру
- Сначала обнови соответствующий раздел в `PROJECT.md` отдельным PR (`docs:`)
- Потом делай код
